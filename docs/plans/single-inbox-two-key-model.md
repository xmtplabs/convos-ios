# Single-Inbox Two-Key Model ‚Äî Identity Local, Backup Key Synced

> **Status**: Draft (rev 1)
> **Created**: 2026-04-29
> **Supersedes**: the "single synced identity" assumption in [docs/plans/icloud-backup-single-inbox.md](./icloud-backup-single-inbox.md) ¬ß"Two keys, two roles ‚Äî don't collapse them"
> **Related**: [ADR 011 ‚Äî Single-Inbox Identity Model](../adr/011-single-inbox-identity-model.md), [docs/plans/single-inbox-identity-refactor.md](./single-inbox-identity-refactor.md)

## TL;DR

The current model stores the user's identity (signing key + database key) in the keychain with `kSecAttrSynchronizable: true`. iCloud Keychain replicates that slot to every Apple-ID-paired device, which by Apple's design is **inherently multi-device**. Combined with the fact that every fresh device that opens the app mints its own MLS installation against that shared identity, we end up with N installations on a single inbox even when the product intent is "only one device active at a time".

The two-key refactor:

1. **Per-device identity** (signing key + databaseKey) ‚Üí stored locally per device, **not synced**. Encrypts the local SQLCipher XMTP DB and signs MLS commits.
2. **Backup key** (a fresh 32-byte secret) ‚Üí stored in keychain with `kSecAttrSynchronizable: true`, **the only synced slot**. Wraps the backup bundle; lets a new device unseal an existing backup and adopt the bundled identity.

After the refactor, opening the app on a second Apple-ID-paired device that has *not* explicitly run Restore does **not** silently mint a new installation against the original inbox. The device sees a backup is available, the user must consciously tap Restore, the destination device adopts the bundled identity, and `XMTPInstallationRevoker` retires the prior installation. Reset on a non-active device wipes the local identity copy without propagating the deletion via iCloud Keychain ‚Äî it leaves the backup key intact so a future restore is still possible. There is no path that auto-registers a new installation against the existing inbox without going through Restore.

This matches the single-device intent without breaking the seamless-restore property.

## Why the current model misses the intent

The "single-inbox identity" promise from ADR 011 was: one inbox per user, with the identity material being the same key set wherever the user has signed in. The implementation chose iCloud Keychain sync as the delivery mechanism for that promise. That mechanism inherently gives every Apple-ID-paired device a copy of the keys.

Concretely:

- `KeychainIdentityStore.identityAccount` is stored with `synchronizable = true`. Two devices on the same Apple ID both load the same `KeychainIdentity` on launch.
- Each device that calls `Client.create` mints a new `installation_id` against that identity. Device A ‚Üí A1 in the inbox state, Device B ‚Üí B1 in the inbox state. Both are valid registrations.
- `XMTPInstallationRevoker.revokeOtherInstallations` retires *previously-registered* installations as part of restore, but does nothing about future registrations from any device that still has the synced keys in keychain.
- `SessionManager.tearDownInbox` ("Reset device") was made to preserve the keychain identity (PR #760 commit `6b08d001`) because deleting it would cascade through iCloud Keychain and brick every other device. The trade: a reset device cannot fully wipe its own copy without taking everyone down with it.

The visible effect: a second device boot mints another installation; restore creates a new one; reset locally is a no-op against the synced identity slot. Even when revocation correctly retires N-1 installations, a device whose local copy is intact will mint installation N+1 on its next "fresh-feeling" launch. The MLS-side pain ‚Äî `Group is inactive`, peer commits required to add new installations, days-long reactivation latency ‚Äî is the symptom of this churn.

This document corrects the storage layer so the symptom doesn't keep recurring.

## The two-key contract

### Identity slot (per-device, **not synced**)

| Field | Source |
|---|---|
| `privateKey` | secp256k1 keypair generated by the device |
| `databaseKey` | 32-byte CSPRNG generated by the device ‚Äî encrypts SQLCipher XMTP DB |
| `inboxId` | derived from `privateKey` (current behavior) |
| `clientId` | per-installation tag (current behavior) |

Stored at keychain account `convos-identity-local`, service `org.convos.ios.KeychainIdentityStore.v4-local`, **synchronizable: false**, accessible: `kSecAttrAccessibleAfterFirstUnlock`.

### Backup-key slot (synced via iCloud Keychain)

| Field | Source |
|---|---|
| `backupKey` | 32-byte CSPRNG, single value |

Stored at keychain account `convos-backup-key`, service `org.convos.ios.KeychainIdentityStore.v4-backup`, **synchronizable: true**, accessible: `kSecAttrAccessibleAfterFirstUnlock`.

### Backup key lifecycle

| Event | Action on local identity slot | Action on backup-key slot |
|---|---|---|
| First launch on Device A (no prior install) | Generate identity, save local | Generate backupKey, save synced |
| First backup write | (no change) | (no change ‚Äî keep existing key) |
| First launch on Device B (Apple-ID-paired, B fresh) | (no local identity yet) | iCloud Keychain delivers backupKey |
| Restore on Device B | After Restore: extract identity from bundle, save to local slot | (unchanged ‚Äî same backupKey kept) |
| `XMTPInstallationRevoker` on Device A's A1 | (no change) | (no change) |
| Reset on Device A (StaleDeviceBanner) | Delete local identity slot | (no change ‚Äî leave backupKey) |
| App launch on A after reset | (slot empty) | iCloud Keychain delivers backupKey ‚Üí bootstrap finds backup ‚Üí user must Restore |
| User taps "Start fresh" on Device A's prompt | (slot stays empty until registration) | Generate new backupKey ‚Üí propagates via sync, A and B now diverge |

### What "Start fresh" must do under the new model

"Start fresh" creates a new identity. Because the previous backupKey is the synced key everyone else was using to read backups, regenerating it on Start-fresh effectively says **"I am a different user from now on, my old backups are gone."** That's the right semantics ‚Äî the synced slot is the only thing that ties devices to the same account, and starting fresh consciously breaks that tie.

The user-facing copy must reflect this: "Starting fresh will replace your account on every device. Old backups become unreadable."

## Threat model

| Concern | Old model | Two-key model |
|---|---|---|
| iCloud Documents leak | Bundle key is `identity.databaseKey` (synced, also encrypts XMTP DB). Leak of bundle + key compromises identity. | Bundle key is `backupKey` (synced). Leak of bundle + key reveals bundle contents (which include identity) ‚Äî same exposure surface. |
| iCloud Keychain leak | Identity (signing + DB key) exposed on every paired device. | `backupKey` exposed; on its own it is useless without the bundle ciphertext. |
| Compromise of one device | Identity leaves with the device ‚Äî readable by attacker. | Same. |
| Multi-device by accident | Yes ‚Äî iCloud Keychain replicates. | No ‚Äî only the backupKey replicates. Identity stays local. |
| Cross-device "restore" | Yes ‚Äî implicit via shared identity. | Yes ‚Äî explicit via Restore flow on bundle + backupKey. |

The exposure surface for the encrypted bundle is the same: bundle ciphertext (in iCloud Documents) + key (in iCloud Keychain). The difference is what *else* the synced key gives an attacker. In the old model, `identity.databaseKey` also unlocks the local XMTP DB on any synced device. In the new model, `backupKey` is bundle-only ‚Äî it doesn't unlock anything on a paired device that hasn't restored.

## Migration

The change has to land cleanly for users currently in production / dev with the old single-synced-identity layout. The migration runs once, before any normal flow uses the keychain identity:

1. **Detect old layout.** A keychain item exists at the old `org.convos.ios.KeychainIdentityStore.v3` account `convos-identity` with `synchronizable: true`.
2. **Generate `backupKey`** if no v4-backup slot exists.
3. **Copy identity to local slot.** Read the v3 synced item, write the same payload to the v4-local slot with `synchronizable: false`.
4. **Delete the v3 synced item.** This propagates via iCloud Keychain to every paired device ‚Äî they will rerun the same migration on their next launch and end up at the same place. *(Crucially this delete is the one we want to propagate; that's why the migration is one-shot and idempotent.)*
5. **Mark migration done** via app-group UserDefaults `convos.keychain.layoutGeneration = "v4-two-key"`.

Edge cases:

- **Mid-migration crash on Device A** (steps 1‚Äì3 done, step 4 not): on next launch, the app sees v3 still present and the local slot already populated; it can safely re-run step 4 (delete the v3 synced slot).
- **Device B launches first under the new build** (before A has migrated): B sees the v3 synced item, copies it to local, deletes the synced item. A's next launch sees the synced item gone but its local slot still empty (because A never ran the migration locally). A's behavior is then: bootstrap finds backup, prompts user to Restore. Equivalent to a fresh-on-paired-device flow.
- **Device B is on the new build, Device A is still on the old build** during a staged rollout: B's migration deletes the synced identity from iCloud Keychain. A on the old build expected the synced identity to be there; loadSync returns nil; A registers a new identity (under the old single-synced model, which still writes synced=true). Now both devices have different identities. **This is the staged-rollout hazard.** Mitigation: gate the migration behind a feature flag tied to "all my paired devices are on the new build" ‚Äî or accept that we ship the migration only after a full version-pin rollout (no skew).

Practical recommendation: don't run the migration automatically until the new build has propagated. Ship the new code with the migration **off** behind `FeatureFlags.shared.twoKeyKeychainMigrationEnabled`, default false in production. Flip after confirming everyone on the team / closed beta is on the new build. This avoids the divergence above.

## Implementation plan

The changes split cleanly across the storage, backup, restore, and session-lifecycle layers. Each step is independently shippable.

### Step 1 ‚Äî Keychain storage layer

- `KeychainIdentityStore` exposes two slot accessors:
  - `loadIdentitySync() throws -> KeychainIdentity?` (local, current `loadSync` semantics)
  - `loadBackupKeySync() throws -> Data?` (synced)
  - `saveIdentity(_:)`, `deleteIdentity()` ‚Äî local slot only
  - `saveBackupKey(_:)`, `deleteBackupKey()` ‚Äî synced slot
- The existing `loadSync()` / `save(...)` / `delete()` API remains available but is internally rewritten to operate on the local slot, so the rest of the codebase can migrate one call site at a time.
- Add a private `KeychainQuery.synchronizable` toggle (already exists per-query ‚Äî just expose it consistently).
- `KeychainIdentityStoreError.identityNotFound` semantics stay the same.

### Step 2 ‚Äî Migration

- New `KeychainLayoutMigrator` enum, runs at app launch from `ConvosClient.client(...)` before `SessionManager` is constructed.
- Reads `FeatureFlags.shared.twoKeyKeychainMigrationEnabled`. If off, no-op.
- If on:
  1. Read app-group `convos.keychain.layoutGeneration`. If `v4-two-key`, no-op.
  2. Read v3 synced slot. If absent ‚Üí set marker, no-op (fresh install on the new code path).
  3. Generate v4 backupKey if missing.
  4. Write v4 local identity if missing.
  5. Delete v3 synced slot.
  6. Set marker.
- Telemetry: `QAEvent.emit(.app, "keychain.migrated_to_v4_two_key")`.

### Step 3 ‚Äî Backup write path

- `BackupManager.createBackup`:
  - Read `backupKey` from new synced slot. If absent ‚Äî fail with `BackupError.backupKeyMissing` (synced slot should always exist post-migration; this is a hard error, not a skip).
  - Read identity from local slot.
  - Bundle's inner metadata gains a new field: `identityPayload: Data` ‚Äî JSON-encoded `KeychainIdentity` so the destination device can adopt the source's identity.
  - Outer AES-GCM seal switches from `identity.databaseKey` to `backupKey`.
  - Inner XMTP archive still uses the per-bundle `archiveKey` (unchanged).

### Step 4 ‚Äî Restore path

- `RestoreManager.restoreFromBackup`:
  - Replace `awaitIdentityWithTimeout` with `awaitBackupKeyWithTimeout` ‚Äî waits for `backupKey` rather than identity. (Identity isn't needed up front; it comes from the bundle.)
  - `decryptAndValidateBundle` uses `backupKey` for outer decrypt.
  - After successful decrypt + validate, extract `innerMetadata.identityPayload`, write it to the local identity slot via `KeychainIdentityStore.saveIdentity(_:)` *before* the `replaceDatabase` step. The XMTP DB swap and subsequent `Client.build` flow then operate against the adopted identity.
- `ConvosRestoreArchiveImporter` is unchanged conceptually but now reads identity from the local slot (which the restore flow just populated from the bundle).

### Step 5 ‚Äî Session lifecycle

- `SessionManager.tearDownInbox` (currently preserves the synced slot per `6b08d001`) now explicitly:
  - Calls `identityStore.deleteIdentity()` ‚Äî wipes the local identity. *Does not* propagate via Keychain because the slot is `synchronizable: false`.
  - Does **not** delete the backup key.
  - Continues to wipe XMTP DBs and DBInbox rows.
- `SessionStateMachine.handleRegister` and `handleAuthorize` operate against the local identity slot (no behavior change once the migration sets it up correctly).
- `BackupCoordinator.resolveBootstrapDecision` doesn't change shape but its semantics tighten: with no local identity and a backup visible, the only valid user choice is Restore (or Start fresh, which now means generating a new backup key as well as a new identity).

### Step 6 ‚Äî Bootstrap settle phase

- `BackupCoordinator.waitForICloudSettle` now polls `identityStore.loadBackupKeySync()` (was `loadSync` for the identity). This is what fresh paired devices should be waiting on.

### Step 7 ‚Äî UI

- `RestorePromptCard` copy reads "A backup from <deviceName> is available on this Apple ID. Tap Restore to set up Convos here, or Start fresh."
- Start-fresh confirmation alert added: "Starting fresh will replace your Convos account on every device on this Apple ID. Old backups become unreadable. Continue?"
- StaleDeviceBanner copy unchanged.
- Backup-debug view shows both slots (local identity present? backup key present? backup key fingerprint?) so QA can verify the layout.

### Step 8 ‚Äî Tests

- `KeychainIdentityStoreTests` covers: separate slot save/load/delete, sync flag round-trips on each slot, idempotent migration.
- `KeychainLayoutMigratorTests` covers: v3-present-only happy path, v3-and-v4-coexist (mid-migration recovery), v4-only short-circuit, marker correctness, mid-migration crash simulated by short-circuiting after step 3.
- `BackupManagerTests` and `RestoreManagerTests` updated stub providers to expose both slots; assert backup is encrypted with backupKey and bundle adoption writes the local identity slot on restore.
- Add a `ResetSemanticsTests` (new) that asserts `tearDownInbox` deletes the local slot and leaves the backup-key slot intact, and that the next launch re-bootstraps via the restore prompt rather than auto-registering.

### Step 9 ‚Äî Rollout

1. Land Steps 1‚Äì8 with the migration feature-flagged off.
2. Internal testing with the flag flipped on closed builds; verify cross-device behavior matches the matrix in ¬ß"Backup key lifecycle".
3. Verify staged-rollout behavior: only one device on new build, others on old ‚Üí confirm no key divergence.
4. Flip the flag for production after the new build has propagated to all devices a typical user has.

## What changes in `tearDownInbox` from PR #760

The recently-shipped guarantee in PR #760 (commit `6b08d001`): *"local Reset preserves the synced keychain identity so other paired devices aren't bricked."* That trade-off was forced by the fact that the only synced slot was the identity itself.

Under the two-key model, that constraint is gone:

- The synced slot is `backupKey` only.
- The local slot is the identity.
- `tearDownInbox` deletes the local slot. No other device cares; their local slots are independent.
- Reset on Device A no longer has to compromise between "wipe my own keys" and "don't brick Device B". It can do both correctly.

The PR #760 comment at `tearDownInbox` becomes moot once Step 5 lands; replace it with the simpler explanation that local Reset only touches local state.

## Open questions

- **Does the inbox-state registry care that a single inbox has both the old (synced-identity-derived) installation and the new (locally-stored-identity-derived) installation post-migration?** No ‚Äî the migration copies the same payload across slots; the `privateKey`-derived `inboxId` is unchanged. The MLS installation IDs come from `Client.create`'s per-DB seed and are unaffected. The migration is an operational shift; XMTP semantics don't move.
- **Can the synced backupKey itself be re-keyed periodically?** Out of scope for v1. If a user wants to rotate, "Start fresh" is the existing path (with the explicit "old backups unreadable" warning).
- **What about App Clip?** `ClipIdentityBootstrap` reads the synced identity today. Under the new model, the clip would either (a) participate in the same two-key flow (clip mints a local identity, reads backupKey from sync if present) or (b) be limited to non-restore scenarios. Defer the choice; the clip currently has narrow responsibilities and likely (a) is straightforward.
- **NSE access?** The Notification Service Extension shares the keychain access group. It needs the local identity (for SQLCipher decryption). The local slot lives in the same app-group access group, so NSE access is preserved. No behavior change.

## Acceptance criteria

The refactor is done when:

1. A fresh install on Device A creates a local identity (synchronizable=false) and a synced backupKey. Verified via `BackupDebugView`.
2. A second fresh install on Device B (Apple-ID-paired) sees a backup available, has *no* local identity yet, and shows the Restore prompt. Tapping "Start fresh" clears the synced backupKey (with explicit confirmation) and proceeds to mint a fresh local identity + new synced backupKey.
3. Restore on Device B adopts the bundled identity, registers a new MLS installation against the existing inbox, revokes prior installations, and lands in `.ready` with all groups syncing.
4. Reset on Device A wipes its local identity. The next launch on A behaves like step 2.
5. `tearDownInbox` no longer needs the "deliberately do NOT call identityStore.delete()" comment; that constraint was an artifact of the single-synced-identity model.
6. The MLS-side reactivation pain on stuck groups is reduced because the system no longer keeps spawning new installations under the same inbox without going through Restore.
