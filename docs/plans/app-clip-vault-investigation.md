# App Clip + Vault Migration — Level of Effort Investigation

## Status: Research / Investigation

## The Question

Can the Vault provide a seamless migration path from App Clip to full app install? When a user opens a Convos invite link, joins via the App Clip, and later installs the full app, can they retain their conversations without re-joining?

## Background

### App Clip Constraints (Apple)
- **50 MB binary size limit** (iOS 16+, thinned for device) — previous limits were 10 MB (iOS 14) and 15 MB (iOS 15)
- App Clips share an **App Group container** with the main app (same `$(APP_GROUP_IDENTIFIER)`)
- App Clips share a **Keychain access group** with the main app
- App Clips can persist data for **up to 8 hours** after last use, then iOS may delete the App Clip's container
- After **30 days** of inactivity, iOS will delete the App Clip and its data
- App Clips **cannot** use iCloud Keychain, CloudKit, or background activities
- App Clips **can** use the shared App Group and Keychain, which survive the full app install

### Current App Clip Target
- `ConvosAppClip` target exists, bundle ID `$(MAIN_BUNDLE_ID).Clip`
- Uses `ConvosCore` (which pulls in XMTP SDK, GRDB, etc.)
- Entry point: `ConvosAppClipApp.swift` — initializes `ConvosClient`, shows `ConversationsView`
- Shares App Group and Keychain access group with the main app
- Currently a near-clone of the full app — not yet scoped down

### The Size Question
The **XMTP SDK (`libxmtp`)** is the primary binary size concern. It includes:
- Rust FFI binary for MLS cryptography (`libxmtp_bindings_swift`)
- Protocol buffer definitions
- Networking stack (gRPC)

A rough estimate: the XMTP binary alone is **~15-25 MB** (arm64, thinned). With the **50 MB limit** (iOS 16+), XMTP likely fits — but the total including ConvosCore, GRDB, Firebase, Sentry, and app code needs to be measured. If the full stack comes in under 50 MB, Scenario A (full XMTP in the App Clip) becomes viable, which dramatically simplifies the migration story.

## Migration Scenarios

### Scenario A: App Clip WITH XMTP (if size allows)

If we could fit XMTP into the App Clip (unlikely without significant SDK changes):

1. User opens invite link → App Clip launches
2. App Clip creates a per-conversation identity (wallet key + XMTP inbox)
3. User joins the conversation, sends/receives messages
4. Identity keys are stored in **shared Keychain** (accessible by main app)
5. XMTP database stored in **shared App Group container**
6. User installs full app
7. Full app reads identity keys from shared Keychain
8. Full app opens XMTP database from shared App Group
9. **Seamless migration** — conversations, messages, and identity are already there
10. Vault is created, App Clip identity is added to it

**Vault role:** Minimal. The shared Keychain + App Group already handle migration. The Vault would only be needed once the user adds a second device.

**LOE:** Low (if size constraint is met). The current architecture already uses shared Keychain + App Group.

### Scenario B: App Clip WITHOUT XMTP (lightweight join flow)

Since XMTP likely won't fit, the App Clip would need a different approach:

**Option B1: Server-assisted join with deferred XMTP setup**
1. User opens invite link → lightweight App Clip launches (no XMTP)
2. App Clip sends a join request via a REST API to the Convos backend
3. Backend holds the join request in escrow
4. App Clip shows a "waiting for approval" UI + prompt to install full app
5. User installs full app
6. Full app picks up the pending join request from the backend
7. Full app creates the XMTP identity and completes the join

**Vault role:** None in the join flow. Standard Vault usage after.

**LOE:** High. Requires new backend endpoints, a server-side escrow system, and breaks the trustless model (the backend would temporarily hold invite context).

**Option B2: Keypair generation in App Clip, XMTP setup deferred**
1. User opens invite link → App Clip launches
2. App Clip generates a secp256k1 keypair locally (no XMTP needed — just `secp256k1` library)
3. App Clip stores the private key in **shared Keychain**
4. App Clip sends the join request DM via a lightweight API (or stores the invite slug + generated keys for later)
5. User installs full app
6. Full app reads the private key from shared Keychain
7. Full app calls `Client.create(account: key)` to bootstrap the XMTP inbox
8. Full app completes the join flow

**Vault role:** The private key generated by the App Clip becomes the conversation identity. The Vault syncs it to other devices normally.

**LOE:** Medium-High. Requires:
- A minimal crypto library in the App Clip (secp256k1 key generation only, ~200KB)
- Saving invite context (slug, keypair) to shared Keychain/App Group
- Main app logic to detect "pending App Clip joins" and complete them
- No messaging possible in the App Clip — just "join request sent, install app to chat"

**Option B3: Preview-only App Clip (no join, just install prompt)**
1. User opens invite link → App Clip shows conversation preview (name, member count, profile photos)
2. App Clip shows "Install Convos to join this conversation"
3. User installs full app
4. Full app opens with the invite deep link and completes the normal join flow

**Vault role:** None.

**LOE:** Low. But provides minimal value — just a branded install prompt.

## Vault as Migration Bridge

The Vault's core value for App Clip migration is **not in the join flow** but in **post-install continuity**:

1. **Shared Keychain is the bridge.** Both App Clip and main app access the same Keychain via the shared access group. Any identity keys the App Clip creates are immediately available to the main app after install. This works today without the Vault.

2. **Vault adds multi-device sync.** Once the main app is installed and the user has conversations, the Vault syncs those keys to other devices. The App Clip → main app transition is a single-device migration (same device), so the Vault isn't needed for that specific transition.

3. **Vault prevents key loss.** If the App Clip's 8-hour data window expires before the user installs the full app, the App Group container data (XMTP databases) may be deleted. But **Keychain data persists** even after App Clip deletion. So identity keys survive, but message history doesn't. The Vault could pre-emptively sync the key to the cloud (via iCloud Keychain, which the App Clip can't do) — but this only works from the main app, not the App Clip.

## Recommended Approach

### Phase 1: Validate binary size (1-2 days)

Before any implementation, measure the actual App Clip binary size:

```bash
# Build App Clip for release (thinned for single architecture)
xcodebuild archive -scheme "ConvosAppClip" -archivePath AppClip.xcarchive \
  -destination "generic/platform=iOS"
# Export with app thinning
xcodebuild -exportArchive -archivePath AppClip.xcarchive \
  -exportOptionsPlist ExportOptions.plist -exportPath AppClipExport
# Check the App Thinning Size Report in AppClipExport/
```

**If under 50 MB → Scenario A is viable.** This is the best outcome — full XMTP in the App Clip with zero migration friction.

**If over 50 MB → strip Firebase + Sentry first**, then remeasure. Those are optional for an App Clip (no crash reporting or App Check needed for a ephemeral experience). This could save ~8-11 MB.

**If still over 50 MB → fall back to B2 or B3.**

### Phase 2: Decide on App Clip scope

| Approach | Binary Size | User Value | LOE | Trustless |
|----------|-------------|-----------|-----|-----------|
| **A: Full XMTP** | ~25-40 MB (maybe fits 50 MB) | **High — full chat in App Clip** | **1-2 weeks** | Yes |
| B2: Keypair + deferred join | Small (~3 MB) | Medium — join request sent, chat after install | 1-2 weeks | Yes |
| B3: Preview-only | Very small (~2 MB) | Low — just an install prompt | 1-2 days | Yes |
| B1: Server-assisted | Small (~3 MB) | Medium | 2-3 weeks | No |

**Recommendation: Validate Scenario A first.** If XMTP fits in 50 MB, the App Clip can offer the full join + chat experience. Migration to the full app is seamless via shared Keychain + App Group. Fall back to B3 → B2 only if the binary doesn't fit.

### Scenario A: Full XMTP App Clip (preferred if size allows)

If XMTP fits, the migration path is trivially simple:

1. App Clip creates identity → stores in **shared Keychain** (already happens via `KeychainIdentityStore`)
2. XMTP database stored in **shared App Group container** (already happens via `AppEnvironment.containerURL`)
3. User chats in the App Clip
4. User installs full app → **everything is already there** (keys, database, messages)
5. Vault is created on first full app launch, App Clip identity is included

**What needs to change for Scenario A:**
- Keep Firebase in the App Clip (needed for App Check — push notifications work in App Clips for up to 8 hours per launch)
- Strip Sentry from App Clip target (reduce size, crash reporting not critical for ephemeral experience)
- Scope down the App Clip UI to conversation join + chat only (no settings, no conversation list, no onboarding)
- App Clip entry point: parse invite URL → join conversation → open conversation view
- Register for push notifications on launch (token valid for 8 hours per session)
- Main app detects existing App Clip data on first launch and adopts it

**Vault role in Scenario A:** The Vault doesn't participate in App Clip → main app migration (same device, shared Keychain handles it). Once the main app creates the Vault, the App Clip's conversation keys are automatically included.

### Fallback: Vault integration with B2

If Scenario A doesn't fit and we go with B2 (keypair generation in App Clip):
- App Clip generates keypair → stores in shared Keychain with a "pending-app-clip-join" tag
- App Clip stores invite slug + conversation context in shared App Group `UserDefaults`
- Main app on first launch checks for pending App Clip joins
- Main app creates XMTP client from the saved keypair
- Main app completes the invite join flow
- Vault picks up the new conversation key normally via `VaultKeyCoordinator.startObservingInboxes()`

The Vault doesn't need any App Clip-specific changes. The existing `KeychainIdentityStore` shared access group handles the key handoff, and `VaultKeyCoordinator` automatically shares new conversation keys to paired devices.

With the 50 MB limit, there's a good chance the full stack fits — especially if Firebase and Sentry are stripped from the App Clip target. **Must measure to confirm.**

## Key Findings

1. **50 MB limit likely makes full XMTP feasible.** Apple's iOS 16+ App Clip size limit of 50 MB should accommodate XMTP + ConvosCore, especially without Firebase/Sentry. This is the highest-value path — must measure to confirm.

2. **The Vault doesn't change the App Clip migration story.** Shared Keychain already bridges App Clip → main app on the same device. The Vault adds cross-device sync but that's a post-install concern.

3. **Shared Keychain is the migration mechanism**, not the Vault. Keys stored by the App Clip in the shared Keychain access group are immediately available to the main app. The existing `KeychainIdentityStore` and `AppEnvironment` already use the shared App Group.

4. **If XMTP fits, migration is nearly free.** The current architecture already stores identity keys in the shared Keychain and XMTP databases in the shared App Group container. The main app would just detect and adopt existing data.

5. **App Group container data is ephemeral** for App Clips (8-hour to 30-day window). Identity keys survive in Keychain, but XMTP message databases may be cleaned up. Users should be nudged to install the full app promptly.
