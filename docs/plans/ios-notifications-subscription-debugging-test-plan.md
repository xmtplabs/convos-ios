## Key Interactions to Verify

Stack 1
- Cold launch with push permission already granted: app foregrounds and DeviceRegistration row in backend has the current APNS token (not the previous one, not absent). Test against a freshly erased simulator.
- Debug View "Register Device Again": tap once, observe a single `/v2/device/register` call in Datadog; nothing creates a new IOSPushNotificationRegistrar (check that PushNotificationRegistrar.token before == after).
- Open an invite, tap Join, leave the app sitting on the joining state for 30s. Datadog must show at most one "Subscribing to topics" log line (versus the ~10 every 30s today). After the join completes, one more reconcile is acceptable.
- Background -> foreground does not trigger an unnecessary `/v2/notifications/subscribe` call if nothing changed.
- Sign out and sign in as a different identity: first reconcile after the new sign-in MUST hit the wire (cache invalidated on identity change).
- Turn airplane mode on, trigger a reconcile, turn airplane mode off, trigger a reconcile: second call MUST hit the wire (cache is not written on subscribe failure).

Stack 2
- Open the Push Registration Probe screen with the app in a known-good state: all rows render with green checks; no raw token appears anywhere on screen.
- Tap "Probe backend registration" while the app is offline: error message renders in place, view stays interactive, no crash.
- Tap "Force topic reconcile" (if shipped this stack): a `/v2/notifications/subscribe` call lands at the backend with `force: true` and bypasses idempotency.
- Tap "Copy diagnostics JSON": output contains hashes only, no raw push token, no raw topic strings, no JWT. Paste into a Slack thread is safe.
- Backend: hit `/v2/notifications/debug/status` with a valid JWT and verify response shape contains only booleans, integers, timestamps, and hashes. No raw token. No raw topic string. No JWT echo.

## Edge Cases

Stack 1
- AppDelegate.didRegisterForRemoteNotificationsWithDeviceToken fires before ConvosApp.init completes — should not happen per SwiftUI lifecycle, but the fix must not crash if `PushNotificationRegistrar.configure()` has not been called. Save path needs a configured-or-noop guard; the existing `shared` accessor's fatalError must not be on the save() hot path. See D2 in review.
- Reconcile during a flaky network: subscribe() throws -> cache is NOT updated -> next reconcile attempts to send. Asserted in unit + an integration test (simulate URLProtocol failure).
- Reconcile during identity rotation: cache cleared before computeDesiredSubscriptions runs.
- Reconcile when APNS token rotates mid-session (rare): hash key includes pushTokenSha256 so cache miss is automatic.
- discoverNewConversations throws: the count is undefined; treat as 0 to be conservative (no reconcile), since a partial failure means we have no positive signal that anything changed.

Stack 2
- /v2/notifications/debug/status with a JWT whose deviceId does not match the request deviceId: 403, no information leaked.
- Snapshot row missing because the user is new: response says `snapshot.exists: false`. UI must render this without showing "mismatch".
- Snapshot row stale because the underlying ClientIdentifier just migrated devices (per D4, this should be deleted in the same tx): debug screen shows "no snapshot" not "stale snapshot".
- subscribe with `kindSummary` totals that don't add up to `topics.length` (client bug): backend logs a warning but does not reject — the data integrity is on the iOS side.
- subscribe with the same hash twice within the idempotency TTL: second call returns 200 with `skipped: true` and does NOT update the snapshot's `lastSubscribeAt` (or update it to a stale-tolerant value — see test plan question for backend).
- Webhook arrives with an installation.id that has no ClientIdentifier: warning logged with 4 fields, response still 200 so XMTP notification server doesn't retry.

## Critical Paths

Stack 1 (regression IRON RULE — see "Test review" in skill)
1. *REGRESSION*: requestDiscovery() called 10x in a row with no new conversations -> AT MOST 1 subscribe API call. This is the bug. Without this test, the fix can regress silently the next time someone touches SyncingManager.
2. *REGRESSION*: reconcile -> subscribe fails -> reconcile -> second call must hit the wire. Without this, a transient failure makes iOS silently drift from backend.
3. *REGRESSION*: Boot -> APNS token arrives -> DeviceRegistration row in backend matches the local token. Existing implicit coverage assumed the property-injection race didn't fire; new test pins behavior to PushNotificationRegistrar.shared.

Stack 2 (regression IRON RULE)
4. *REGRESSION*: /v2/notifications/subscribe with identical (deviceId, clientId, topicHash, apnsEnv, pushTokenHash) within TTL skips the XMTP server calls. Without this, idempotency silently regresses.
5. *REGRESSION*: /v2/device/register that migrates a clientId to a new deviceId deletes the matching NotificationSubscriptionSnapshot row in the same transaction. Without this, debug-status calls report false token mismatches for the lifetime of the snapshot row.
6. *REGRESSION*: successful subscribe writes the snapshot row; failed subscribe leaves `lastRemoteApplySucceeded: false` or writes nothing. Without this, the debug screen shows lies.
