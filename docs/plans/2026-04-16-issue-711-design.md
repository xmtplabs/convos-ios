# Issue #711 — Move non-standard runners off paid GitHub Actions to WarpBuild

## Background / Problem

The repository incurred **~$601 in paid GitHub Actions costs last month**. GitHub provides complimentary Actions usage for open-source projects on standard runners (`ubuntu-latest`), but charges premium rates for non-standard options — in this repo, the macOS runners (`macos-latest`, `macos-26`).

We need to migrate every job currently running on a non-standard (paid) GitHub runner to a WarpBuild-hosted equivalent so the project stops accruing paid-runner minutes on macOS builds/tests.

## Requirements (EARS)

- **R1** — WHEN any of the four paid-runner jobs runs, THE SYSTEM SHALL execute on a WarpBuild macOS runner (`runs-on:` label prefixed with `warp-macos-`) instead of a GitHub-hosted `macos-*` runner.
- **R2** — WHEN `swiftlint.yml` runs on WarpBuild, THE SYSTEM SHALL successfully install SwiftLint and run `swiftlint lint --strict` with the same behavior as before.
- **R3** — WHEN `convoscore-tests.yml` `unit-tests` and `integration-tests` jobs run on WarpBuild, THE SYSTEM SHALL have Xcode 26 (or the version the project requires) available so `swift --version` / `xcodebuild -version` succeed and `./ci/run-tests.sh` passes.
- **R4** — WHEN `keychain-identity-store-tests.yml` runs on WarpBuild, THE SYSTEM SHALL be able to run `xcodebuild test ... -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.1'` (same simulator destination as today).
- **R5** — WHERE a workflow file is migrated, THE SYSTEM SHALL retain the job's existing triggers, concurrency groups, caches, env vars, artifacts, and failure behavior unchanged.
- **R6** — WHERE a migration is complete, no `runs-on` label in the repo SHALL be `macos-latest`, `macos-26`, `macos-*` (any macOS-hosted GitHub runner), or any non-standard Ubuntu label (`ubuntu-latest-4-cores`, etc.). `ubuntu-latest` remains allowed because it is free for OSS.

## Non-Requirements

- No changes to the scripts invoked by the workflows (`./ci/run-tests.sh`, `./ci/fly/deploy`, etc.).
- No changes to tests themselves.
- No changes to `ubuntu-latest` jobs — they are already on the free tier.
- No infrastructure setup on the WarpBuild side (the repo owner must have the WarpBuild GitHub App installed on `xmtplabs/convos-ios`; if not, the jobs will queue indefinitely). If WarpBuild is not yet installed, the app owner should install it from https://github.com/apps/warpbuild.

## Design

### Chosen WarpBuild label: `warp-macos-latest-arm64-6x`

Per WarpBuild's published runner catalogue, the available macOS labels are:

- `warp-macos-latest-arm64-6x` — latest supported macOS, M4 Pro, 6 vCPU, 14 GB RAM
- `warp-macos-14-arm64-6x` — macOS 14 Sonoma
- `warp-macos-13-arm64-6x` — macOS 13 Ventura

The repo's current runners use `macos-26` (macOS 26 Tahoe, host for Xcode 26) because the iOS target is 26.0. `warp-macos-latest-arm64-6x` tracks the newest macOS WarpBuild supports, which is the closest equivalent. If WarpBuild's "latest" lags behind GitHub's `macos-26`, the migrated jobs may need an explicit Xcode selection step (e.g. `sudo xcode-select -s /Applications/Xcode_26.app`) — added pre-emptively where needed so Xcode 26 is selected when available.

### File-by-file changes

1. **`.github/workflows/swiftlint.yml`** — change `runs-on: macos-latest` → `runs-on: warp-macos-latest-arm64-6x`. SwiftLint install/run steps unchanged.

2. **`.github/workflows/convoscore-tests.yml`**
   - `unit-tests.runs-on: macos-26` → `warp-macos-latest-arm64-6x`
   - `integration-tests.runs-on: macos-26` → `warp-macos-latest-arm64-6x`
   - `deploy-backend` and `cleanup` remain `ubuntu-latest`.

3. **`.github/workflows/keychain-identity-store-tests.yml`** — `runs-on: macos-26` → `warp-macos-latest-arm64-6x`. The `Select Xcode (use default on macOS 26)` step currently just runs `xcodebuild -version`; if the WarpBuild image ships with multiple Xcodes or a non-26 default, this step should explicitly select Xcode 26 via `sudo xcode-select -s /Applications/Xcode_26.app` — we will adjust only if CI fails post-migration. For now we leave the existing `xcodebuild -version` step as a diagnostic, which will reveal the available Xcode on the first run.

### What we are NOT touching

- `ubuntu-latest` runners — free on the OSS tier.
- Xcode-selection logic — left as-is. The first CI run post-migration will surface whether `warp-macos-latest-arm64-6x` ships Xcode 26 by default. If not, a follow-up pin (e.g. `sudo xcode-select -s /Applications/Xcode_26.app`) is a one-line change — out of scope for this PR unless the first run fails.
- Caching paths — identical between GitHub and WarpBuild macOS images (`~/Library/Caches/org.swift.swiftpm`, etc.), so existing cache steps are unaffected.

## Testing & Validation

This change cannot be validated locally — it must be validated by running the workflows in CI. After merging:

1. **Gate: `swiftlint.yml`** — pushes to `main` / `dev` or any PR should trigger the workflow; it must run to green on `warp-macos-latest-arm64-6x`.
2. **Gate: `convoscore-tests.yml`** — the `unit-tests` job must run and pass on the new runner. `integration-tests` must pick up the ephemeral backend and pass.
3. **Gate: `keychain-identity-store-tests.yml`** — triggered by changes to keychain paths; the `xcodebuild test` destination `iPhone 17,OS=26.1` must resolve (i.e. the runner must have Xcode 26 + the iOS 26.1 simulator runtime). If not, a follow-up PR adds explicit Xcode / simulator install.
4. **Cost gate (post-merge observation)** — billing for paid Actions minutes on subsequent month should drop to near-zero for macOS.

If any of gates 1–3 fail on the first CI run, remediation is straightforward (add `sudo xcode-select -s /Applications/Xcode_26.app` and, if needed, `xcrun simctl runtime add` for the simulator runtime). This PR starts minimal to avoid over-correcting before observing real runner behaviour.

## Open Questions for the User

1. **Is WarpBuild already installed on `xmtplabs/convos-ios`?** If not, the WarpBuild GitHub App must be installed and the org enabled before the renamed `runs-on` labels will pick up runners. Without it, jobs will queue forever. Please confirm this is set up (or install at https://github.com/apps/warpbuild).
2. **Is `warp-macos-latest-arm64-6x` the right size?** 6 vCPU / 14 GB is WarpBuild's standard tier. Xcode builds sometimes benefit from larger instances; if builds are slow, we can upgrade to a larger tier in a follow-up.
3. **Do we want to pin the macOS version?** Using `warp-macos-latest-arm64-6x` tracks whatever WarpBuild considers latest. If reproducibility matters, we could pin to an explicit version (e.g. `warp-macos-15-arm64-6x`) once we know WarpBuild supports it.

Proceeding with the minimal migration (label swap only) while awaiting confirmation on these questions.
