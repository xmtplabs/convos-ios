Identity system, end to end.

Follow one user from the moment they first launch the app to the moment a push notification wakes the NSE. The path crosses five surfaces — the keychain, SessionManager, SessionStateMachine, the invite join flow, and the NSE delivery cache — each documented in detail in its own ADR.

01

First launch — fresh install

No identity in the keychain yet

The app launches with an empty keychain slot and an unbound XMTP identity. SessionManager.loadOrCreateService() is the only call that needs to happen — its synchronous one-slot cache decides whether to authorize or register based on what's in the keychain.

01 App ConvosAppDelegate boots; SessionManager is instantiated.
02 Session loadOrCreateService() takes the lock, calls KeychainIdentityStore.loadSync(); returns nil — no identity.
03 Session AuthorizeInboxOperation takes the .register branch.Generates secp256k1 key pair + 256-bit SQLCipher key, writes both to the shared keychain slot.
04 Session SessionStateMachine builds the XMTPiOS.Client with deviceSyncEnabled: true and the full codec list.Local SQLCipher database file xmtp-<host>-<hash>.db3 is created under the app group.
05 App Onboarding carousel runs; the user picks a name + avatar; DBMyProfile is created.Subsequent .ready transitions on new conversations will propagate this via activate-sync (see ADR 005).
02

Returning launch — identity already present

The hot path

Every subsequent launch — same device, or a new device signed into the same Apple ID — sees the identity in the keychain. The path through SessionManager is identical; only the branch in AuthorizeInboxOperation changes.

iCloud sync. The keychain item has kSecAttrSynchronizable = true, so the identity follows the user across their Apple ID devices. A second device's first launch hits the .authorize branch — deviceSyncEnabled on the XMTP client then replays group memberships and message history from the XMTP history server.
03

Via the App Clip

Clip → keychain → main app

When a user taps an invite link without the main app installed, iOS launches the App Clip. The Clip's job is to write the identity into the shared keychain slot so the main app can pick it up on first launch — onboarding is skipped, the user lands directly in the conversation they joined.

01 Clip User taps invite link. iOS launches ConvosAppClip.
02 Clip ClipIdentityBootstrap builds SessionManager in .clipBootstrap mode — push registration, asset renewal, prewarm, and timers are skipped.
03 Clip loadOrCreateService() forces materialization — register branch runs, identity is written to the shared keychain slot.
04 Clip The Clip processes the invite slug (verify signature, decrypt token, DM join request).See ADR 001.
05 User User installs the full app from the Clip card.
06 App First main-app launch: loadOrCreateService() finds the identity, takes the .authorize branch, lands directly in the joined conversation.
04

Joining a conversation

How an invite consumes the singleton identity

The invite system (ADR 001) uses the user's single inbox for both ends of the handshake. The inviter's inbox signs the slug; the joiner's inbox sends the JoinRequest DM and ends up as a member of the resulting MLS group. There's no per-conversation inbox provisioning — the join flow reuses the singleton service from SessionManager.

Pending invites are tracked in PendingInviteRepository with no clientId scoping — the protocol's only no-arg method is allPendingInviteDetails(). Under single-inbox, the question "which inboxes have pending invites?" collapses to "the user has some, or doesn't" so the API has one bool.

05

Push delivery

Backend ↔ APNs ↔ NSE

The backend never sees the inbox ID. It routes by a random clientId generated for the user at registration time. Push payloads carry ciphertext only — the NSE decrypts in-process, reads the singleton identity, and validates the payload's clientId against the stored one before reusing its cached service.

01 XMTP A peer sends a message; XMTP network notifies the backend that a delivery is pending.
02 Backend Looks up the clientId for the recipient; forwards an APNs push with the encrypted XMTP payload.The backend has no inbox ID, no plaintext, no membership context.
03 NSE iOS wakes NotificationService. CachedPushNotificationHandler reads the singleton identity from the keychain.
04 NSE Validates payload clientId vs. stored clientId.Mismatch → tear down the cached service and rebuild for the new identity (e.g. user signed out + in during the same NSE process).
05 NSE Cached MessagingService decrypts the MLS message and surfaces the notification body.Cache TTL is 15 minutes from last access.
06

Account wipe

deleteAllInboxes()

The wipe is the only path that removes the keychain identity. It runs via deleteAllInboxes(), which keeps the cached service alive through the entire teardown so a concurrent loadOrCreateService() (a push arriving mid-delete, a SwiftUI sync accessor) observes the being-torn-down service rather than racing a fresh one against the same SQLCipher files.

01 · DENY Mark every conversation Consent denied on each MLS group so resync can't re-create them.
02 · LEAVE Leave groups Best-effort leaveGroup on every owned conversation.
03 · CLEAR Delete keys + DB Wipe keychain slot, delete the SQLCipher file, drop GRDB rows.
04 · UNREG Tell the backend unregisterInstallation(clientId:) so push routing stops.

The NSE shares the keychain. If the wipe happens in the main app while the NSE is foreground, the NSE drops the next delivery (no identity) and best-effort unregisterInstallation(clientId:)s its clientId.
