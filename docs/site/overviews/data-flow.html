Overview · 02

Data flow & persistence.

GRDB sits at the centre. Writers mutate it (and emit XMTP messages); repositories read it (and emit Combine streams). SwiftUI subscribes. The XMTP SDK never escapes the messaging seam — everything else talks to XMTPClientProvider.

01

Shape

Writers in, repositories out

GRDB is the single source of truth for what the UI shows. Writers are the only code that puts rows into GRDB. Repositories are the only code that hands GRDB rows out. SwiftUI observes the repositories. There is no read path that bypasses GRDB — even fresh XMTP stream events flow through a writer before becoming visible.

02

One write, end to end

Example: user sends a message
01 View User taps send. ViewModel calls OutgoingMessageWriter.send(...).
02 Writer Writes the message row to GRDB with deliveryStatus = .pending.The row is visible immediately — the UI shows the message in flight.
03 Writer Calls XMTPClientProvider.send(...) on the conversation's MLS group.Awaits the SDK's commit; converts SDK errors to domain errors before re-raising.
04 Writer On success: updates the row to delivered. On failure: failed, with a retry timestamp.
05 Repo MessagesRepository's Combine publisher fires; the ViewModel updates; SwiftUI re-renders the row in its new state.
The first write is always local. Optimistic state is a first-class concept: pending rows are visible before XMTP has acknowledged. The writer reconciles back to delivered / failed rather than waiting before writing.
03

Three layers of model

DB row → hydrated → domain

Three flavours of type live under Storage/. Keeping them separate means GRDB-friendly representations (codable, flat) stay structurally distinct from the domain types SwiftUI consumes (computed properties, denormalized convenience fields).

LayerFolderRole
Database Models Storage/Database Models/ DBConversation, DBMessage, DBMyProfile etc. — Codable + GRDB-typed; raw row shape.
Hydration Storage/Hydration/ Joins one or more DB rows into a denormalized fetched object (e.g. DBConversationDetails reads the conversation plus its members and last message in one query).
Domain Models Storage/Models/ Conversation, Profile, Message — what the UI consumes. Includes computed properties (display name, default emoji from ADR 007, lock state derivation, etc.).

Repositories

Sole owners of observed reads

Each repository wraps one query (or one cluster of related queries) and emits a Combine publisher. ViewModels subscribe; SwiftUI views observe the ViewModel. Writers never call repositories — they write rows, and the publisher refresh is GRDB's job.

07

XMTP seam

XMTPClientProvider — the only protocol that knows about XMTPiOS

Every call into XMTP goes through XMTPClientProvider. The protocol mirrors the SDK's surface but is the only place that imports XMTPiOS types — every writer, every repository, every test uses the protocol. Production builds inject the real provider; tests inject a mock. Even debug-time tooling lives behind extensions on the protocol (XMTPClientProvider+MetadataDebug, XMTPClientProvider+HiddenMessagesDebug) rather than reaching for SDK types directly.

Rule. If you find yourself writing import XMTPiOS anywhere outside ConvosCore/Messaging/ or ConvosCore/Inboxes/, that's a code smell. The fix is either to extend the provider protocol or to introduce a higher-level adapter.
08

Stream side

XMTP → StreamProcessor → writer

Inbound XMTP messages are dispatched through StreamProcessor. It routes each message by content type to the appropriate writer: plain messages to IncomingMessageWriter, profile updates to the profile path, explode signals to the explosion writer, capability requests to the capability handler, and so on. Each writer is the single owner of its row updates.

09

Testability

Why the seam matters

Three deliberate choices make the data layer test-friendly without needing the iOS Simulator:

  • Cross-platform ConvosCore. The package compiles on macOS, so unit tests run under Swift's test runner with no simulator boot.
  • XMTPClientProvider mocks. The test target injects an in-memory provider; writers exercise their real GRDB code against a real MockDatabaseManager.
  • Writers are protocol-typed at call sites. Every writer used by a ViewModel is reached through a protocol, so tests can swap in fakes per-domain without rebuilding the world.
