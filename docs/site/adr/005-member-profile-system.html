ADR 005 · Profiles

Member profile system.

Each user has a single global profile on device. It seeds every new conversation, and propagates into existing conversations the next time each one becomes active. On the wire, profiles are still per-conversation XMTP messages — the global profile never leaves the device.

Accepted · Last reviewed 2026-05-21

01

Shape of the system

Global vs. per-conversation

Conceptually, profiles live in two places. The global profile (DBMyProfile) is the user's single source of truth — the name and avatar they expect to show by default. The per-conversation profile (DBMemberProfile, one row per (conversationId, inboxId)) is what peers actually see, and what the wire formats serialize. The global profile never leaves the device.

Two storage tiers
Local only DBMyProfile · global Raw image bytes, content digest, asset id, metadata, updatedAt. Never transmitted.
On the wire DBMemberProfile · per (conversationId, inboxId) Name, encrypted avatar URL + salt + nonce, MemberKind, metadata, imageSourceContentDigest.
Forward-looking. The per-conversation row carries an imageSourceContentDigest field. When that field is nil, the row is treated as a per-conversation override (the user explicitly picked a different photo just for this conversation) and is not overwritten by activate-sync. The UI doesn't surface that path yet — the field exists so the model can absorb a future "nickname / different photo per conversation" feature without a schema change.
02

Wire messages

Two XMTP content types, both silent

Profile data travels as XMTP group messages, not via the conversation's shared appData field. Two codecs do all the work, both with shouldPush = false so they don't surface a notification.

CodecContent typeSent whenPayload
ProfileUpdateCodec convos.org/profile_update · v1.0 A member changes their own profile. Optional name, encrypted image ref, MemberKind, metadata. Empty payload clears the profile.
ProfileSnapshotCodec convos.org/profile_snapshot · v1.0 After group creation, after addMembers, after accepting a join request. Every current member's profile in one envelope — solves MLS forward-secrecy gaps for new joiners.

A sender's inbox ID is taken from the XMTP message envelope, never from the payload — impersonation is structurally prevented. The encrypted avatar reference is just {url, salt, nonce}; the key lives in the group's encrypted metadata (see ADR 009).

03

Global profile

DBMyProfile · local-only

One row in DBMyProfile, keyed by inboxId. The image lives as raw bytes; a SHA-256 over the bytes (imageContentDigest) is the canonical identity used to detect changes. The imageAssetIdentifier is the PHAsset id, used only to preselect the system photo picker.

FieldTypeRole
inboxIdStringPrimary key (one row per identity).
nameString?Display name.
imageDataData?Raw avatar bytes (encrypted at rest by SQLCipher).
imageContentDigestString?SHA-256 base64 — the sync identity for the avatar.
imageAssetIdentifierString?PHAsset id; preselects the picker, not a sync key.
metadataProfileMetadata?JSON map of typed key-value pairs.
updatedAtDateUsed for ordering and for skipping no-op syncs.
04

Activate-sync

Propagating the global profile on demand

When a conversation transitions to .ready, ConversationStateManager.scheduleProfileSync hands the work to ProfileSyncCoordinator — an actor keyed by conversationId so two syncs for the same conversation cannot race. The coordinator runs MyProfileWriter.syncFromGlobalProfile:

01 Sync Compare global.name to the per-conversation member.name.If different, send a ProfileUpdate with the new name.
02 Sync Compare global.imageContentDigest to member.imageSourceContentDigest.If they differ, re-upload the new avatar from the global bytes and stamp the new digest into the row.
03 Sync If the global avatar was cleared and the member still has one, propagate the removal.An override row (imageSourceContentDigest == nil) is left alone.
04 Stream Peers process the ProfileUpdate via StreamProcessor and upsert their DBMemberProfile row.UI observes the GRDB change and re-renders the member's name and avatar.

06

Snapshots

Closing the forward-secrecy gap

MLS forward secrecy means a fresh joiner may not be able to decrypt older messages, including past ProfileUpdates. A ProfileSnapshot sent after the joiner is added contains every current member's profile in one envelope — they get the full picture without needing to replay history.

ProfileSnapshotBuilder scans up to 500 messages (descending) to assemble the snapshot, applying the resolution table above for each member. Snapshots are sent at three points: group creation, after addMembers, and after an invite is accepted.

07

Encrypted avatars

AES-256-GCM, key lives in the group

Avatars are resized, JPEG-compressed, then encrypted with AES-256-GCM under a key derived from the group's imageEncryptionKey via HKDF with a per-image salt. The ciphertext is uploaded to S3; the wire message carries only {url, salt, nonce}. The key itself never travels in ProfileUpdate or ProfileSnapshot — it lives in the conversation's encrypted metadata.

Loading is handled by EncryptedImageLoader with a three-tier cache (memory → disk → network), keyed by SHA-256 of the ciphertext for deduplication. See ADR 009 for the full encryption story.

08

Privacy model

Current posture
PropertyGuarantee
Global-profile detection by peers The global profile is local-only. Peers only see the per-conversation ProfileUpdate; they can't tell it came from a preset rather than typed in.
Profile scope Visible only to members of that conversation. There is no profile server.
Anonymity Profiles are optional; users can participate with no name or avatar set.
Wire-level encryption Both codecs use XMTP message encryption; the avatar ciphertext is independently AES-GCM encrypted.
Cross-conversation linkability Per-conversation rows are independent, but the underlying inboxId is shared across the user's conversations (see ADR 011). A peer who sees the user in two conversations can correlate via that ID.
