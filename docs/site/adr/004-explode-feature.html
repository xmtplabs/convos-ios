ADR 004 · Conversations

Explode.

A creator can permanently dissolve a group. The mechanism is remove-all-then-leave at the MLS group layer: send the ExplodeSettings message, remove every other member, then leave. Either trigger — the message or the MLS "removed" commit — is enough for a receiver to drop the conversation locally.

01

What explode is

A protocol-layer dissolution, not a key wipe

Explode runs entirely at the MLS group layer. The creator broadcasts a custom ExplodeSettings content type with a push payload, removes every other member from the group, then leaves it. After the leave commit, there is nothing left of the group on the network: no members, no admin, no commit head, nothing to resync against. Locally, every participant's expiresAt filter hides the conversation from the UI.

Keychain material is never touched. The guarantee is "messages are gone from the local database of every participant that applied the commit", not cryptographic finality.

02

Sender flow

ConversationExplosionWriter.explodeConversation

The call order is pinned by tests and reached through the ExplodeGroupOperationsProtocol seam so the MLS operations are unit-testable without an XMTP fixture.

01 Creator Read currentInboxId for the gating check (only the creator can explode).
02 Creator Call sendExplode(expiresAt:) — sends the ExplodeSettings message with shouldPush = true.Sent first so offline members get a push notification even if the subsequent removal fails.
03 Creator Update local DBConversation.expiresAt to now.The conversation immediately drops out of the UI via ConversationsRepository's expiresAt == nil || expiresAt > Date() filter.
04 Creator metadataWriter.removeMembers(…) on every other member.Emits MLS GroupUpdated events so peers who missed the ExplodeSettings still see they were removed.
05 Creator group.leaveGroup() — the creator exits last.Fallback: if leaveGroup() throws, denyConsent on the conversation so it can't resync.

04

Receiver flow

Two triggers, one outcome

A receiver hides the conversation as soon as either signal arrives. The creator fires both precisely so a lost push or an out-of-order delivery still converges via the MLS commit stream.

On ExplodeSettings

IncomingMessageWriter.processExplodeSettings writes expiresAt to the row. If the timestamp is in the past, it posts .conversationExpired for immediate cleanup. If it's in the future (scheduled explode), it posts .conversationScheduledExplosion so the worker can arm a timer. Either way, the UI's expiresAt filter only hides the row once expiresAt <= now.

On the MLS "removed" event

When the current inbox appears in the commit's removedInboxIds, conversation.postLeftConversationNotification() fires. The UI removes the row from the list. Same end state as the message path, just a different trigger.

05

ExpiredConversationsWorker

The reaper

The worker is the single owner of cleanup for expired rows. It listens for four signals and re-runs the cleanup query on each: app foreground, .conversationExpired, .conversationScheduledExplosion, and .explosionNotificationTapped (the user tapped the explode push).

The cleanup query is the soft-delete filter inverted — expiresAt <= now rows are eligible for hard deletion of associated messages, members, profiles, and invites. Under single-inbox identity (ADR 011) the worker never touches the keychain or the XMTP database; it scopes deletions to the conversation row and its dependencies.

What is & isn't touched

Single-inbox scope rules
SideAction
TouchedThe conversation row, its messages, members, profiles, invites, and any local state keyed to it.
TouchedMLS group state on the network — removed members + creator leave commit.
Not touchedKeychain identity. There is exactly one, shared with every other conversation.
Not touchedThe XMTP database file — it still holds other conversations.
Not touchedThe DBInbox row — there's only one and it represents the user.
07

Scheduled explode

Future-dated expirations

The scheduleExplosion path sends an ExplodeSettings with a future expiresAt and otherwise leaves the group in place. When the timer fires in ExpiredConversationsWorker, the local app posts .conversationExpired and the conversation is filtered out of the UI. The creator does not leave the group on the network at that point; the scheduled path currently relies on every member applying expiresAt locally.

08

Trust model

What the mechanism does and doesn't enforce
ThreatMitigation
Honest peers fail to deleteNone needed — the MLS remove + leave commits make the group state unrecoverable on the network.
Offline peer misses the explodePush notification (shouldPush = true) wakes them; on next app launch the worker reconciles either trigger.
Out-of-order deliveryEither trigger (message or MLS remove) is sufficient; whichever arrives first wins.
Duplicate explode messagesIdempotency check: rows already with expiresAt set are not re-stamped.
Malicious peer keeps ciphertextNot enforceable — see "What is & isn't touched". Encrypted message content may exist in third-party backups or modified clients.
System caches retain message textNot enforceable — iOS may keep notification text briefly after deletion.
