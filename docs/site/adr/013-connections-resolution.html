ADR 013 · Connections

Resolution & picker.

One agent contract fronts both connection subsystems. The user picks a provider per conversation; the resolver remembers; the manifest tells the agent which provider serves which verb in which conversation. The device and cloud subsystems still own their execution paths.

Accepted · 2026-05-04 · Last reviewed 2026-05-21

01

Shape

A thin coordination layer with four jobs

The resolver doesn't execute anything. Device invocations still flow through ConvosConnections; cloud calls still go agent-side via Composio. The resolver only persists the user's per-conversation choice and routes the device path through CapabilityInvocationRouter when a request arrives.

Four responsibilities
Taxonomy CapabilitySubject + ProviderID A flat enum of user-facing subjects and a dotted-string provider namespace shared across both subsystems.
Registry CapabilityProviderRegistry In-memory actor of currently usable providers. Populated by each subsystem at bootstrap and on link/unlink.
Routing table GRDBCapabilityResolver Maps (subject, conversationId, capability) → Set<ProviderID>. Survives restart.
Contract capability_request / _result Agent asks; user answers; manifest republishes. Same wire shape for device and cloud answers.

Subjects & providers

The flat enums users and agents share

CapabilitySubject

A user-facing taxonomy, separate from ConnectionKind. Not every subject has a device counterpart (.tasks, .mail); not every ConnectionKind is a subject (.motion is a sensor primitive, not something to "grant access to").

SubjectFederates reads?

ProviderID

Opaque dotted-string identifiers — device.calendar, composio.google_calendar. The resolver routes by registry lookup, never by parsing the prefix. Two helpers exist for the rare cases where the namespace matters: ProviderID.cloudServiceId (strips composio.) and ConnectionKind.fromDeviceProviderId(_:) (inverse for device.<kind>).

Resolver

capabilityResolution table

GRDBCapabilityResolver is the production implementation; InMemoryCapabilityResolver mirrors the surface for tests. Validity of any resolution is enforced by CapabilityResolutionValidator at every write site.

FieldRole
subjectOne of the 10 cases above.
conversationIdPer-conversation routing.
capability.read, .write_create, .write_update, .write_delete.

The table has ON DELETE CASCADE from the conversation row, so deleting a conversation removes its resolutions automatically.

04

Wire types

capability_request / _result
CodecDirectionContent type
CapabilityRequestCodecagent → userconvos.org/capability_request/1.0
CapabilityRequestResultCodecuser → agentconvos.org/capability_request_result/1.0

Both encode their Codable Swift type as JSON, consistent with the ConnectionInvocation codecs from ADR 012. Both reject decode of a higher version than they understand; CapabilityRequest truncates rationale to 500 chars and preferredProviders to 16 entries on decode so a hostile sender can't bloat the picker. CapabilityRequestResult always replies — .approved / .denied / .cancelled — so the agent can correlate by requestId and stop waiting. The reply's providers field carries what was actually persisted, not what the agent asked for, so an agent that supplied a hint can detect whether it was honored.

05

Picker variants

CapabilityPickerLayout.Variant

CapabilityRequestHandler.computeLayout picks one of five layouts based on the linked-provider count and the federation × verb shape. The handler picks; the view renders.

VariantWhen
.confirmExactly one linked provider — default-approve card with "Use a different one?" disclosure.
.singleSelectMultiple linked providers; single-select. Used for any write verb or read on a non-federating subject.
.multiSelectMultiple linked providers; multi-select. Only renders for .read on a federating subject.
.connectAndApproveZero linked providers — card doubles as a "Connect a calendar" entry point.
.verbConsentSame subject already has a resolution for some other verb; new verb defaults to those providers and the card is a consent prompt (no picker).

preferredProviders from the agent is honored only when every preferred id is in the linked set; otherwise the picker falls through to its default. Preferences never override federation rules — a write verb is single-select even if the agent asks for two providers.

06

Invocation router

CapabilityInvocationRouter

Every ConnectionInvocation arriving from an agent flows through the router before reaching ConnectionsManager. The router consults the resolver and either dispatches or replies with a structured status.

01 Router No subject mapping for the invocation's kind → reply unknownAction.
02 Router No capability declared for the action → reply unknownAction.
03 Router Empty resolution for (subject, capability, conversationId) → reply capabilityNotEnabled.The agent should send a capability_request first.
04 Router Resolution contains device.<kind> → dispatch to ConnectionsManager and return its slice of the result.
05 Router Resolution exists but contains only cloud providers → reply executionFailed instructing the agent to call those provider APIs directly.Cloud federation is the agent's concern, not the device's.
07

Manifest

CapabilityManifest on member-profile metadata

After a resolution changes — or a provider's link state flips — the host writes a CapabilityManifest to profile.metadata["connections"] on the next ProfileUpdate. This replaces the cloud-only payload from ADR 012 with a unified shape that carries both device and cloud entries. Older readers that expected the cloud-only shape skip device entries (they're additive).

Provider lists are sorted alphabetically by id.rawValue so equivalent state produces byte-identical JSON, avoiding spurious ProfileUpdate writes when nothing has changed.

Per-device only. Cloud grants persist on the member profile (canonical, sync via XMTP) and the resolver row persists on the device only — there's no cross-device sync of the routing table today. A user who linked Google Calendar on iPhone and switches to iPad sees the link, but not the per-conversation routing; they re-pick on first request.
