ADR 008 · Storage

Asset lifecycle & renewal.

Every S3 object expires 30 days after upload. The client renews profile and group images every ~15 days by re-uploading; chat images are never renewed and fall off the cliff. No prefixes, no tags — asset type is invisible in URL structure.

Accepted · Last reviewed 2026-05-21
01 Shape 02 Renewal 03 API contract 04 Expired recovery 05 Tradeoffs 06 Code map
01

Shape of the system

One bucket, one rule

The bucket has a single lifecycle rule: every object expires after 30 days from upload, measured at the S3 layer via LastModified. No prefix or tag is used to classify asset type, so a URL gives away nothing about whether it points at a chat image, a profile avatar, or a group photo. The only way to keep an object alive past 30 days is to upload its ciphertext again as a new object.

Lifecycle by asset class
Chat images Never renewed Fall off after 30 days. Conversations remain readable; the inline image becomes a placeholder.
Profile avatars Renewed ~ every 15 days Stay alive while the user opens the app at least once per renewal window.
Group images Renewed ~ every 15 days Same cadence as avatars; renewal is keyed off the asset, not the conversation.
02

Renewal

AssetRenewalManager.performRenewalIfNeeded()

The renewal manager runs best-effort on app launch and on a timer while the app is foreground. Each pass collects the URLs that need attention via AssetRenewalURLCollector, then submits them in a batch to the backend. Failure is non-blocking — renewal never gates the rest of the app from coming up.

01 Client performRenewalIfNeeded() reads defaultRenewalInterval = 15 days and decides whether to run.Skipped if the last successful pass was inside the interval.
02 Client AssetRenewalURLCollector walks the renewable assets (profile avatars + group images) and emits their S3 keys.
03 Client ConvosAPIClient.renewAssetsBatch(assetKeys:) → POST /v2/assets/renew-batch.Renewal is a fresh ciphertext re-upload server-side, not a metadata stamp on the existing object.
04 Server Returns AssetRenewalResult { renewed, failed, expiredKeys }.expiredKeys lists the keys the server couldn't find — those have already aged out.
05 Client For each expired key, ExpiredAssetRecoveryHandler clears the local reference (UI falls back to placeholder) and, if the source bytes are still in the local cache, re-uploads to mint a fresh URL.

04

Expired recovery

When the bytes still live locally

A confirmed not_found means the S3 object aged out before the renewal pass reached it. The recovery handler clears the stored URL so the UI falls back to placeholder (an emoji avatar, the deterministic group emoji from ADR 007), then checks the local cache. If the cleartext bytes are still there, the handler re-encrypts and re-uploads, then stamps the new URL on the owning row. If the bytes are gone, recovery ends — the user is prompted to re-pick an image the next time they open the relevant settings screen.
