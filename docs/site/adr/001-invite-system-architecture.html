ADR 001 · Invites

Decentralized invite system.

An invite is a URL-safe slug carrying a signed protobuf payload. The joiner verifies the signature, decrypts the wrapped conversation token, and DMs the inviter a join request. No backend ever sees the conversation ID.

Accepted · Cryptographic redesign tracked in docs/plans/invite-system-single-inbox.md · Last reviewed 2026-05-21

01

Anatomy of an invite

What ends up inside the URL slug

A slug encodes a SignedInvite protobuf: an InvitePayload plus the inviter's secp256k1 signature over it. The payload includes the encrypted conversation token, the inviter's inbox ID, the invite tag, and optional metadata (name, description, image URL, expiration, single-use flag).

SignedInvite · protobuf wire format
Payload InvitePayload Invite tag · encrypted conversation token · creator inbox ID · optional metadata.
Signature secp256k1 ECDSA · 65 bytes 64-byte signature + 1-byte recovery ID. Lets the joiner recover the inviter's public key.
Encoding URL-safe Base64 · asterisks every 300 chars DEFLATE applied when the result is smaller. Compression marker byte 0x1F.
02

Invite tag

The 10-character handle the conversation carries

Every conversation has a persistent invite tag — a 10-character random alphanumeric string (regex ^[A-Za-z0-9]{10}$) stored in the conversation's XMTP custom metadata. The same tag is embedded in every invite for that conversation, so it does four jobs at once:

  • Joiner protection. When XMTP delivers the "you've been added" event, the joiner compares the conversation's tag to the one their invite carried. A mismatch means someone is trying to add them somewhere they didn't agree to go.
  • Conversation identity binding. The tag binds the encrypted conversation token to a specific conversation; transplant attacks fail this check.
  • Revocation. Updating the tag in metadata invalidates every existing invite at once.
  • Duplicate join detection. The joiner queries the local database for the tag before sending a request; if a conversation already exists, the join is short-circuited.
03

Slug pipeline

From conversation ID to URL-safe string
01 · ENCRYPT Conversation token Encrypt the conversation ID with ChaCha20-Poly1305 under a per-inviter derived key. HKDF-SHA256 · salt "ConvosInviteV1" · info "inbox:<inboxId>" · AAD = inviter inbox ID
02 · PACK InvitePayload Build the protobuf: tag, encrypted token, inviter inbox ID, optional metadata. version · nonce(12B) · ciphertext · auth_tag(16B)
03 · SIGN secp256k1 ECDSA Sign SHA-256 of the serialized payload; emit 65 bytes (signature + recovery). Public key is recovered later from the signature — never carried in the slug.
04 · ENCODE URL-safe slug DEFLATE if smaller, then URL-safe Base64; insert * every 300 chars for iMessage. Decompressed-size cap: 1MB invite / 10MB metadata to block decompression bombs.
04

Sharing

QR · Universal Link · Messages

The slug is the only thing that travels. It sits at the tail of the Convos universal link (or in a QR code, or pasted into any messenger). Receivers either tap the link — the main app or the App Clip handles the URL — or paste it into the join field. Universal-link routing is documented in the module map overview; the bytes in transit are the same.

05

Join handshake

Joiner ↔ inviter, no third party

The join is a one-round DM. The joiner sends the inviter a JoinRequest message carrying the original signed invite. The inviter's client verifies, then adds the joiner to the MLS group. The joiner's client confirms the conversation it lands in by comparing invite tags.

01 Joiner Decode slug, verify signature, recover inviter's public key.Decrypt the conversation token using the HKDF-derived key.
02 Joiner Check local DB for a conversation with this invite tag.If present — short-circuit, open the existing conversation.
03 Joiner Open or reuse a DM to the inviter; send JoinRequest with the signed invite as payload.
04 Inviter Verify the signature and creator inbox ID against the user's identity; check expiration and single-use flags.
05 Inviter If valid — group.addMembers(…) the joiner. If invalid — DM-block via XMTP consent.Spam protection collapses to "valid invites get added, invalid ones lose the DM thread."
06 Joiner XMTP delivers the "added to group" event. Joiner compares the group's invite tag to the one in their invite.Tag mismatch is treated as a malicious add — surfaced to the user, not silently accepted.
06

Error feedback

InviteJoinError — sent back through the same DM

When a join request fails for legitimate reasons (not spam), the inviter's client sends an InviteJoinError message back through the join DM. The codec follows the same custom-content-type pattern as ExplodeSettings. Each error carries the original inviteTag so the joiner's state machine can match it to the pending request and transition to .joinFailed with the right user-facing copy.

ConditionError typeUser-facing meaning
Conversation expired / deletedconversationExpired"This invite is no longer valid."
Conversation not foundconversationNotFound"That conversation can't be located."
Inviter has denied consentconsentNotAllowed"This invite can't be accepted."
Any other failuregenericFailure"Something went wrong — try again."

Delivery is fire-and-forget with shouldPush: true so the joiner gets the error even when their app isn't foreground. Unknown error types — fields a newer build might emit — fall back to conversationExpired on parse, keeping older builds graceful.

Threats & mitigations

Current posture
ThreatMitigation
Invite forgerysecp256k1 ECDSA verification, public key recovered from the signature.
Token reuse across identitiesAAD binding in ChaCha20-Poly1305 ties the token to a single inviter.
Conversation ID exposureID encrypted under an HKDF-derived key; only the inviter can decrypt without the slug.
Malicious "added to group" eventJoiner compares invite-tag from slug to the group's metadata tag before trusting the add.
Timing attacks on verificationConstant-time comparison on cryptographic operations.
Decompression bombs1MB / 10MB decompressed-size caps.
Spam / fake join requestsXMTP consent: invalid requests trigger DM block.
Replay attacksOptional invite expiration and single-use flags in the payload.
08

Planned cryptographic redesign

What changes next, and why
Forward-looking. The invite payload includes the inviter's inbox ID — and since identity is global (one inbox per user), a decoded slug reveals the inviter's global identifier to anyone who picks it up. The redesign that addresses this — a different join-request transport and a token payload that doesn't expose the inviter — is tracked in docs/plans/invite-system-single-inbox.md. The wire format described on this page is the live format today; the redesign will land as a deliberate breaking change with a versioned slug header.
