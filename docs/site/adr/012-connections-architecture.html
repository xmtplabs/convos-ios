ADR 012 · Connections

Connections architecture.

Two subsystems under one umbrella. ConvosConnections wraps device data sources (HealthKit, EventKit, Photos, …) into Swift types agents talk to over XMTP. CloudConnections brokers OAuth grants for services agents call directly. ADR 013 fronts both behind one resolver.

Accepted · 2026-04-28 · Last reviewed 2026-05-21

01

Shape

One umbrella, two subsystems, one adapter

Agents reach into the user's life in two structurally different ways. Device data sources live on the user's iPhone, gated by per-framework iOS authorization, and are reached via the device. Cloud OAuth services live on third-party servers, gated by an OAuth grant, and are reached around the device — the agent calls the third-party API directly through Composio. The wire shape and the execution path are materially different; the user-facing language ("connect your calendar") is the same.

Packages
Device ConvosConnections Top-level Swift package. Cross-platform (iOS + macOS). No XMTP dependency.
Wire ConvosConnectionsXMTP Separate package under ConvosCore. The only place that knows about both ConvosConnections and XMTPiOS.
Cloud CloudConnections (in ConvosCore) Lives in ConvosCore because grants persist in GRDB and ride on member-profile metadata.
02

Device subsystem · ConvosConnections

Sources, sinks, and the manager

The device subsystem is a manager that registers two kinds of handler per ConnectionKind: a source that observes the iOS framework and emits payloads, and a sink that accepts invocations from agents and translates them into framework calls. Authorization is per-source — every framework has its own consent UI and the subsystem doesn't try to homogenize them.

ConnectionsManager fan-outs sensor payloads to every conversation enabled for that source's read capability, gating writes by an EnablementStore keyed by (kind, capability, conversationId). ScreenTime sits in a separate library product (ConvosConnectionsScreenTime) because the Family Controls entitlement needs Apple approval; apps without that entitlement build the rest of the package alone.

03

XMTP adapter

Three codecs, JSON over XMTP
CodecDirectionContent type
ConnectionPayloadCodecdevice → agentconvos.org/connection_payload/1.0
ConnectionInvocationCodecagent → deviceconvos.org/connection_invocation/1.0
ConnectionInvocationResultCodecdevice → agentconvos.org/connection_invocation_result/1.0

All three encode their Codable Swift type as JSON. JSON over protobuf because the agent runtime's natural form is JSON tool calls, schema evolution is easier than fixed field numbers, and the payloads stay human-readable for debugging. XMTPConnectionDelivery implements ConnectionDelivering through an injected conversation-lookup closure, so the device package never sees ConvosCore's GRDB storage. XMTPInvocationListener filters by content type, gates schema versions, and replies with a synthetic executionFailed for version skew it doesn't understand — so an agent gets a structured "no" rather than silence.

04

Cloud subsystem · CloudConnections

Composio-brokered OAuth

A cloud connection is an OAuth grant the user authorizes once per service per device. The grant itself is not transmitted as an XMTP message — once the OAuth flow completes via ConvosAPIClient.completeCloudConnection, a CloudConnectionGrantEntry is appended to the sender's CloudConnectionsMetadataPayload and re-published as part of the next ProfileUpdate. Agents observe grants by reading the member profile under the connections key (see ADR 013 for the unified manifest shape that supersedes the cloud-only payload).

01 Agent Send connection_grant_request (convos.org/connection_grant_request/1.0). Renders as a "Connect Google Calendar" card.
02 User Tap "Open Settings" → OAuth flow brokered by Composio.
03 Client completeCloudConnection stores the grant in GRDB and emits a fresh ProfileUpdate with the new entry on connections metadata.
04 Agent Reads the manifest from the sender's member profile and calls Composio directly using composioEntityId / composioConnectionId.The device is not in the loop for cloud calls.

06

Discovery

How agents know what to ask for

Agents don't discover capabilities at runtime. They have a priori knowledge of action schemas because the convos-cli vendors the same domain types as TypeScript / Bun equivalents. The fallbacks for version skew are structured: unknown action name → unknownAction; newer schemaVersion than the device knows → executionFailed with a version-mismatch message; capability not enabled for the conversation → capabilityNotEnabled.

The unified discovery surface — a capability_request / capability_request_result pair plus the CapabilityManifest published on ProfileUpdate — is the subject of ADR 013.
