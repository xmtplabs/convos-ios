ADR 011 · Identity

Single-inbox identity model.

Every Convos user has exactly one XMTP inbox. Its identity material lives in one keychain slot, shared with the Notification Service Extension and the App Clip via the app group. Push routing keeps a clientId indirection so the backend never sees the inbox.

01

Identity boundary

Who reads the singleton slot

Three processes share the user's identity: the main app, the Notification Service Extension (NSE) that decrypts pushes, and the App Clip that handles deep-link invite previews. They all read the same keychain entry, gated by the app-group access group; no other process gains visibility.

App group · keychain access group
Main app Convos.app Drives onboarding, registers the identity if absent, owns the live MessagingService.
Extension NotificationService.appex Reads identity to decrypt MLS payloads. Validates clientId per delivery.
App Clip ConvosClip.appex Registers identity into the slot so the main app can pick it up after install.
One entry · service org.convos.ios.KeychainIdentityStore.v3 · account convos-identity
02

Keychain slot

Attributes & API surface

KeychainIdentityStore owns exactly one slot. Its public surface is save, load, loadSync, delete — no multi-identity API and no clientId-keyed lookups.

AttributeValueWhy
kSecAttrService org.convos.ios.KeychainIdentityStore.v3 Versioned so a future identity format can coexist without overwriting.
kSecAttrAccount convos-identity Single fixed key — there is only one identity to look up.
kSecAttrAccessible …AfterFirstUnlock NSE needs the identity to decrypt pushes after first unlock; …ThisDeviceOnly would block iCloud sync.
kSecAttrSynchronizable true Identity follows the user across Apple ID devices via iCloud Keychain.
kSecAttrAccessGroup App-group identifier Shared with the NSE and the App Clip; no other code reads it.

The stored blob is the secp256k1 private key (XMTP identity) plus the 256-bit database encryption key used by the XMTP SDK to encrypt the local SQLCipher file. Both rotate together; there is no separate database key lifecycle.

03

XMTP client

One client per user, configured by SessionStateMachine

SessionStateMachine.clientOptions constructs one XMTPiOS.Client per user. deviceSyncEnabled is on so a second device signed into the same Apple ID replays group memberships and history from the XMTP history server. The local XMTP database lives under the app-group container as xmtp-<gRPC-host>-<hash>.db3, encrypted with the per-install database key from the keychain.

04

Push routing

backend ↔ APNs ↔ NSE

The backend sees only a random clientId per user; it never learns the XMTP inbox ID, and ciphertext never leaves the XMTP network. The NSE re-derives the routing target on every delivery, with a short-lived identity-keyed cache that rebuilds the moment the identity changes.

NSE delivery cache

CachedPushNotificationHandler keeps one MessagingService at a time, tagged with the (inboxId, clientId) pair it was built for. A 15-minute stale-by-access TTL bounds idle memory. Every delivery follows the same checklist:

01 NSE Read singleton identity from keychain.If missing — best-effort unregisterInstallation(clientId:) and drop the delivery.
02 NSE Validate payload clientId against the stored clientId.If mismatch — tear down the cached service and rebuild for the new identity.
03 Cache On match, the cached MessagingService decrypts and surfaces the notification.Cache TTL = 15 minutes from last access; bounded by NSE process lifetime.
05

Session lifecycle

SessionManager.loadOrCreateService()

SessionManager is a synchronous one-slot cache guarded by OSAllocatedUnfairLock<MessagingService?>. The first call loads the keychain identity through loadSync() and builds a service via AuthorizeInboxOperation; every subsequent call returns the cached instance.

01 · ENTER loadOrCreateService() Take the lock, return cache if hot.
02 · LOAD loadSync() Read identity from keychain inside the lock.
03 · BUILD AuthorizeInboxOperation .authorize when identity exists, .register when not.
04 · CACHE Return + memoize Store the live service in the slot; release the lock.

When the load fails

If loadSync() throws — daemon error, corrupt JSON, iCloud sync mid-flight — the lock caches a FailedIdentityLoadOperation-backed service that surfaces the real error through sessionStateManager.currentState. Registering on top would overwrite a potentially recoverable identity, so the next call retries loadSync() inside the lock instead. On success the errored cache is replaced with a real service; on continued failure the same frozen instance is returned with no rebuild thrash.

Account wipe

deleteAllInboxes() keeps the cached service live through teardown so a concurrent loadOrCreateService() (a push arriving mid-delete, a SwiftUI sync accessor) observes the being-torn-down service rather than racing a fresh one against the same SQLCipher files. The cache is cleared only after the XMTP client, the keychain entry, and the DBInbox rows are gone.

06

App Clip handoff

Clip writes once, app picks up after install

The App Clip target declares the same keychain-access-groups entitlement as the main app and instantiates KeychainIdentityStore against the same access group. ClipIdentityBootstrap builds a SessionManager in .clipBootstrap mode — push registration, asset renewal, prewarm, and worker timers are all skipped — then forces one service materialization so the register path writes identity into the shared slot.

01 Clip User taps an invite link. ClipIdentityBootstrap spins up.
02 Clip SessionManager in .clipBootstrap mode materializes a service, taking the .register path.Identity is written to the shared keychain slot.
03 User User installs the full app from the clip card.
04 App First launch: loadOrCreateService() finds the identity, takes the .authorize branch.Onboarding carousel is skipped; user lands on the conversation the clip joined.
07

Conversation explode

Remove-all-then-leave at the protocol layer

Since identity is shared across every conversation, destroying the inbox is not a per-conversation primitive. Explode runs at the MLS group layer: the creator sends an ExplodeSettings message, removes every other member with group.removeMembers(…), then calls group.leaveGroup().

Receivers drop the conversation locally on the ExplodeSettings message or on the MLS "removed" commit, whichever arrives first. No keychain material is touched. The guarantee is "messages are gone from the local database of every participant that applied the commit" — not cryptographic finality. See ADR 004 for the receiver-side handling and the consent-fallback path.

08

Threat model

Current posture per category
ThreatPostureHow
Backend learns user's XMTP identity Protected Backend only sees the per-user clientId; the inbox ID is never sent.
Backend correlates user across conversations Not protected One clientId per user; correlation across conversations on the backend is intrinsic.
Peer compromise leaks cross-conversation content Not protected One key per user — a compromised inbox decrypts every conversation it participates in.
Device theft (pre-unlock) reads messages Protected Keychain item is …AfterFirstUnlock; DB is SQLCipher-encrypted with the keychain-stored DB key.
iCloud account compromise leaks identity Not protected kSecAttrSynchronizable is on; an iCloud-account compromise reaches the identity.
NSE serves stale cached service across identities Protected CachedPushNotificationHandler invalidates on (inboxId, clientId) mismatch and stops the prior service before rebuilding.
Forward-looking. A per-conversation override hook is preserved in the schema for a future "nickname / different photo per conversation" feature — the single-inbox layer doesn't preclude that UX, it just doesn't ship it today. See ADR 005 for the profile system the override would live in.
