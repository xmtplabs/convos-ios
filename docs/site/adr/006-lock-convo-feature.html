ADR 006 · Conversations

Lock convo.

A super-admin can close a conversation to new joins. Locking does two things at once: rotate the invite tag so every previously-shared invite becomes unusable, and set the MLS addMemberPolicy to .deny so no member — admin or otherwise — can add anyone.

01

What lock does

Two enforcement layers, one toggle

Locking attacks the problem on two layers. The MLS permission policy is the cryptographic backstop — every conforming client refuses to add a member when addMemberPolicy == .deny. The invite-tag rotation is the defense in depth — any invite slug shared before the lock now references a tag that no longer matches the conversation, so the join request's tag-check fails on the inviter's client before any add is attempted.

02

Lock flow

ConversationMetadataWriter.lockConversation
01 Admin Confirm the action via the lock-confirmation sheet.The toggle is gated to superAdmin by ConversationViewModel.canToggleLock.
02 Writer rotateInviteTag() on the group's custom metadata.Every existing invite for this conversation now references a stale tag.
03 Writer updateAddMemberPermission(newPermissionOption: .deny).MLS permission policy is replicated to every conforming client.
04 Sync On the next sync, ConversationWriter derives isLocked = (permissionPolicySet().addMemberPolicy == .deny) and writes it to DBConversation.UI observes the GRDB change; the lock icon replaces the share icon in the toolbar.
03

State on the row

DBConversation.isLocked

isLocked is a persisted column on DBConversation, derived from the MLS permission set during sync. It isn't a separate source of truth — it's a denormalized cache so SwiftUI observers can react to lock state without doing async XMTP calls. The MLS policy remains the authority; if a client somehow sees an out-of-date isLocked, the next sync corrects it.

04

Unlock semantics

Old invites stay dead
Important. Unlocking flips addMemberPolicy back from .deny but does not re-rotate the invite tag. Invites shared before the lock remain invalid. Users must share a new invite after unlocking — this is by design, so that the lock's "invalidate everything shared so far" property cannot be undone by accident.
05

Who can toggle

superAdmin only

Only members with the superAdmin XMTP role can lock or unlock, matching the gating used for other irreversible actions (remove members, explode). The check is enforced in ConversationViewModel.canToggleLock, which reads isCurrentUserSuperAdmin from the conversation's member roster.
