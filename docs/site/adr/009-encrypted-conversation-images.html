ADR 009 · Storage

Encrypted conversation images.

Every avatar and group image is AES-256-GCM ciphertext on S3. One encryption key lives in the conversation's metadata; each image has its own salt and nonce. The wire reference is just { url, salt, nonce } — the key never travels.

Accepted · Last reviewed 2026-05-21

01

Layout

Where keys, salts, and ciphertext each live

Three distinct things participate in each encrypted image. Keeping them physically separate is what gives non-members no way to decrypt even if they get the ciphertext URL.

Three locations
Group metadata imageEncryptionKey One 32-byte key per conversation, in the encrypted MLS metadata. Visible only to members.
Wire message EncryptedImageRef · {url, salt, nonce} Per-image. Travels in ProfileUpdate, ProfileSnapshot, and conversation metadata.
S3 Ciphertext object Random key. No metadata or prefix that hints at owner, conversation, or asset type.
02

Encrypt pipeline

From raw image to ref
01 · RESIZE Compress to JPEG Cap dimensions and quality before encryption to keep ciphertext bounded.
02 · DERIVE HKDF-SHA256 Mix group key with a random 32-byte salt to derive the per-image AES key. key = HKDF(group.imageEncryptionKey, salt)
03 · ENCRYPT AES-256-GCM Encrypt JPEG bytes with a fresh 12-byte nonce; tag is bundled into the ciphertext. EncryptedPayload { ciphertext, salt, nonce }
04 · UPLOAD S3 PUT + Ref Upload ciphertext, then emit EncryptedImageRef { url, salt, nonce } into the message.
03

Decrypt pipeline

EncryptedImageService.loadAndDecrypt(url:salt:nonce:conversationId:)

Reads run in reverse. The service uses conversationId to look up the group's imageEncryptionKey from MLS metadata, mixes it with the ref's salt via HKDF to recover the per-image key, then decrypts with the nonce. AES-GCM's tag verifies integrity; a tampered byte fails the tag check and surfaces as a load error.

04

EncryptedImageRef

The wire shape
FieldTypeWhy
urlStringS3 location of the ciphertext.
saltData (32 bytes)HKDF input — different per image, so two images encrypted under the same group key use different per-image keys.
nonceData (12 bytes)AES-GCM nonce — uniquely paired with the per-image key for that single encryption.
Note. The encryption key is never carried in the ref. A leaked ref + leaked ciphertext URL is still useless without access to the conversation's encrypted metadata.
05

Scope & rotation

What the v1 key covers

One imageEncryptionKey per group covers every avatar and the group image; per-image salts keep derived keys distinct. The key is not rotated when members are removed — a former member who recorded the key while a member could still decrypt images uploaded before they were removed. Future image authorization will lean on XMTP metadata access controls rather than a key rotation.

06

Threats

What this design defends against
ThreatMitigation
Non-member reading ciphertext from S3No key, no read — the group key isn't on the object or in the ref.
Tampering with the ciphertextAES-GCM tag check fails; load surfaces an error.
Per-image key reuseRandom per-image salt → HKDF derives a fresh key per image even under one group key.
URL leak hints at asset typeNo prefix, no tag (see ADR 008).
Removed member retains decrypt abilityNot mitigated in v1 — key isn't rotated on remove.
